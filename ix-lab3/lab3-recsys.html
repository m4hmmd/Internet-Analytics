def key_bias(rdd):
    # Cumulate the rating values for each key
    cum_rdd = cumulate_by_key(rdd)

    # Count the number of ratings for each key
    count_rdd = count_by_key_rdd(rdd)

    # Join the two sets by their keys
    joined_rdd = cum_rdd.leftOuterJoin(count_rdd).sortByKey()
    
    # Calculate the user bias for each user
    lst_key_bias = joined_rdd.map(lambda a: (a[0], a[1][0] / a[1][1])).collect()
 
    # Finally, change the list to a dictionary
    dict_key_bias = change_list_dict(lst_key_bias)

    return dict_key_bias

Ex 3.7 Evaluation¶

Implement a function error() that takes an RDD containing (userId, movieId, rating) triplets and computes the error (3.2) with respect to the validation set.

  • Use this function to evaluate the baseline predictions that you computed previously.
  • You should obtain an error of approximately 0.867.
In [45]:
# Calculates an error
# used for evaluation of the recommender systems
def error(predictions, validation):
   # Attach keys to values in the prediction and validation sets
    predictions_by_user_product = predictions.map(lambda r: ((r[0], r[1]), r[2]))
    validations_by_user_product = validation.map(lambda r: ((r[0], r[1]), r[2]))

    # Join the two sets by their keys (user, product)
    joined_by_user_product = predictions_by_user_product.join(validations_by_user_product)

    # Calculate the square of the differences for each pair of validation and prediction
    # and re-key by user
    sq_differences_by_user = (joined_by_user_product
        .map(lambda x: (x[0][0], (x[1][0] - x[1][1])**2)))

    # Take the squareroot of the mean of the square differences from above
    # on a user-by-user basis to get each user's error
    user_errors = (sq_differences_by_user
        .map(lambda x: (x[0], (x[1], 1)))
        .reduceByKey(lambda a, b: (a[0] + b[0], a[1] + b[1]))
        .map(lambda x: np.sqrt(x[1][0] / x[1][1])))

    # Finally, take the mean of the each user's error
    mean_error = user_errors.mean()

    return mean_error

Comparison of $\lambda_{tiny} = 10^{-4}$ vs $\lambda_{default} = 10^{-2}$:

The model using a lower regularization parameter of just $10^{-4}$ instead of the original $10^{-2}$ results in a higher validation error. This is because removing regularization increases the model's effective degrees of freedom. This increase in flexibility in turn causes the overfitting of the model to the training data, making it less accurate with the "new" validation data because it is "fit to the noise".

Comparison with $\lambda = 10.0$:

The model using a higher regularization parameter of a whopping $10.0$ rather than the original $10^{-2}$ also results in a much, much higher validation error. This is because overregularizing the model significantly decreases the model's effective degrees of freedom. This lack of flexibility means that the model will be overly generalized and cannot capture the intricacies of details in the data, resulting in the lower validation accuracy.

In the case of a $\lambda$ as large as $10.0$, the regularizer term becomes the dominant term in much of the training loss function, meaning that training itself converges slower and is less effective.

# Procedure to iterate on lambda values
def iterate_on_lambda(iterations, initial_step_factor, diminish_rate):
    for _ in range(iterations):
        iteration = len(sec_322_lambdas) - 2
        print()
        print('===== Iteration {} ====='.format(iteration))
        print()
        print('lambda Error gradient:')
        print('{:5f} - {:5f}'.format(sec_322_validation_errors[-1], sec_322_validation_errors[-2]))
        print('---------------------')
        print('{:5f} - {:5f}'.format(sec_322_lambdas[-1], sec_322_lambdas[-2]))
        grad = (sec_322_validation_errors[-1] - sec_322_validation_errors[-2]) / (sec_322_lambdas[-1] - sec_322_lambdas[-2])
        print('= {:5f}'.format(grad))
        # Like a learning rate
        sec_322_LAMBDA_step_factor = initial_step_factor * diminish_rate**iteration
        print()
        print('step factor is {:5f}'.format(sec_322_LAMBDA_step_factor))
        delta_lambda = -np.sign(grad) * sec_322_LAMBDA_step_factor
        print()
        print('lambda will be adjusted by {:5f}'.format(delta_lambda))
        new_lambda = sec_322_lambdas[-1] + delta_lambda
        print()
        print('old lambda = {:5f}; new lambda = {:5f}'.format(sec_322_lambdas[-1], new_lambda))
        print()
        print('training and evaluating new model...')
        _, _, validation_error = train_and_evaluate(new_lambda)
        print('trainging and evaluation complete.')
        print()
        print('old validation error = {:5f}; new validation error = {:5f}'.format(sec_322_validation_errors[-1], validation_error))
        print('with an improvement of {:5f}'.format(sec_322_validation_errors[-1] - validation_error))
        sec_322_lambdas.append(new_lambda)
        sec_322_validation_errors.append(validation_error)

# Procedure to go back to a previous lambda value and discard future iterations
def go_back_to_iteration(n):
    sec_322_lambdas = sec_322_lambdas[:n]
    sec_322_validation_errors = sec_322_validation_errors[:n]

Concerning User 138494's Recommendations (Ourselves!)

Some of the ratings are decent, others not so much. The lack of accuracy can be attributed to us only rating 11 movies. In collaborative filtering, we are sort of facing the cold-start problem. With only 11 movies rated, it is difficult to find users with similar tastes to us because a large amount of users could be similar to us in the rating of those 11 movies, but have vastly different tastes in the thousands of other movies.

Another contributor to the lack of accuracy is the model itself. An error of 0.8 is not too great.

Finally, it could be that our movie tastes are a significant outlier compared to other users, which would cause the recommender to provide irrelevent recommendations.

    val = val * 100

Observations of the Rank-2 Model

As with the recommendations in the previous section, the results of this model are difficult to interpret. It is not obvious what semantic meanings to give each of the two axes we see above.

It appears that movies which have lower Rotten Tomatoes scores have larger values in the first principle axis, but this trend is not strong since there are highly rated movies everywhere in the plot. There do not appear to be other trends among ratings nor genres.

There are also some extremely tight pairs of movies: for example "Edge of Tomorrow" and "Santa Sangre", and "Control" and "American Pie". However, these movie pairs do not make too much sense.

Besides the tight pairs of movies, there exist other clusters as well, but these clusters do not seem to have semantic meaning either.
